## Key Questions
1. 小程序是否采用单一代码库承载学生端和老师端，还是拆为两个独立小程序应用？
2. 管理后台是否需要首期就支持多校区隔离与细粒度 RBAC？
3. 消息、文件存储、内容审核、服务通知等第三方能力的选型优先级如何安排？
4. 首期交付目标更偏向“能跑通比赛主流程”，还是“直接上全功能版”？

## Decisions Made
| Decision | Rationale |
|----------|-----------|
| 当前先做总体规划，不直接开始编码 | 项目体量极大，必须先明确边界、优先级和里程碑 |
| 需求来源以 `重新师生系统.docx` 为准 | 当前项目为空，文档是唯一正式输入 |
| 规划输出分为架构层、模块层、实施层三层 | 便于后续先确认方向，再进入具体设计与开发 |
| 设计文档和实施计划统一落到 `docs/plans/` | 便于后续持续迭代与追踪 |
| 当前按实施计划的第一批任务执行 `Task 1-3` | 先把工程底座和后端最小服务跑起来 |
| 当前按实施计划的第二批任务执行 `Task 4-6` | 先把三端骨架和账号基础域搭起来 |
| 当前按实施计划继续完成 `Task 7-12` 的 MVP 闭环 | 先把比赛主链路、积分和基础运营能力跑通，再进入持久化与真实集成 |
| 当前优先把 `auth`、`competition`、`registration` 切到真实持久化 | 这三个领域是后续签到、作品、评分、积分链路的基础依赖 |
| 当前已完成 `checkin`、`submission`、`score`、`points` 的真实持久化 | 核心比赛闭环已全部具备 Spring 环境下真实落库能力，可进入接口联调阶段 |
| 当前优先把前端“核心可见页面”接到真实 API | 先让用户和管理员能直接看到真实比赛、结果、积分和评审数据，再继续推进表单类流程联调 |
| 当前表单型页面优先复用现有最小后端命令模型 | 先把比赛发布、报名、签到、作品上传接通，再视需要扩展更完整的报名资料字段 |

| 当前优先补齐小程序登录态持久化与接口级闭环回归测试 | 让真实登录后的体验可持续、可恢复，并把比赛主链路锁进自动化回归 |
| 当前明确移除所有第三方依赖能力，以及积分商城和多校区模块 | 保持项目范围可控，聚焦本地可运行、可验收的剩余 PRD 能力 |
| 当前优先完成剩余模块的 `Task 1`：账号与个人中心基础版 | 这是比赛增强、消息中心和后台治理之前最基础的用户能力补齐 |

## Errors Encountered
| Error | Attempt | Resolution |
|-------|---------|------------|
| 并行读取提取文件时出现时序冲突 | 1 | 确认提取文件已生成后再进行后续读取，改为串行校验 |
| `mvn test` 首次失败，缺少 Spring Boot 启动类与 `ApiResponse` | 1 | 按 TDD 补齐最小应用骨架和统一响应体后恢复通过 |
| Git 检查状态时提示 `dubious ownership` | 1 | 暂未修改全局 Git 配置，后续如需提交再处理 `safe.directory` |
| `pnpm` 在当前盘符上默认链接安装失败 | 1 | 根目录新增 `.npmrc`，并在 `miniapp` 使用 `--ignore-workspace --node-linker=hoisted --package-import-method=copy` |
| 小程序 `uni-app` 脚本直接调用 `vite` 构建失败 | 1 | 改为使用 `uni -p h5` / `uni build -p h5` |
| 并行执行两个 Maven 命令导致本地 `.m2` 缓存锁冲突 | 1 | 改为串行执行 Maven 验证 |
| PowerShell 会把 Maven 的 `-Dtest=A,B` 解析成参数列表 | 1 | 为 `-Dtest` 参数整体加引号后恢复正常 |
| 多个新测试初次失败是因为比赛时间窗口设置不合法 | 1 | 调整测试前置数据为“可报名且已开赛”的合法场景 |
| 引入持久化后测试环境需要 MySQL 兼容数据库能力 | 1 | 新增 `application-test.yml`，使用 H2 MySQL 模式配合 Flyway 自动迁移 |
| 结果查询在数据库分支里丢失了积分值 | 1 | 通过 TDD 新增断言，补充 `V6__score_points_column.sql` 和 `score.points` 持久化读写 |
| Windows 下大块补丁写入触发 `文件名或扩展名太长` | 1 | 切换为多次小步 `apply_patch`，按骨架、逻辑、样式拆分写入 |
| `ProfileApiTest` 初版把学生账号硬编码为 `userId=2`，全量测试时顺序变化导致失败 | 1 | 改为按 `studentNo=S20260001` 动态查询真实用户 ID，恢复全量稳定性 |
| 后台与老师端原有“发布比赛”页面只支持新建，不支持管理存量比赛 | 1 | 增加比赛管理列表、草稿、编辑、推荐/置顶、下架接口与页面交互，完成生命周期增强 |
| Windows 下单次大补丁写入消息模块时再次触发 `文件名或扩展名太长` | 1 | 按 `windows-large-file-writes` 的 staged write 方式拆成多次 `apply_patch`，分段落地迁移、实体、服务和页面 |
| 收紧 `@MapperScan` 后，`FeedbackMapper` 因缺少 `@Mapper` 导致 Spring 上下文启动失败 | 1 | 为 `FeedbackMapper` 显式补上 `@Mapper`，并保留“仅扫描带注解 Mapper”的配置，避免服务接口被误注册为 MyBatis Mapper |
| 审核接口测试首轮在 `MockMvc` 手工解析响应体时出现中文字符集比较异常 | 1 | 为审核接口显式返回 `application/json;charset=UTF-8` 后恢复稳定 |
