# 师生比赛管理平台总体设计

## 1. 项目定位

本项目是一个面向校园场景的多角色比赛管理平台，终端包括：

- 微信小程序：学生端、老师端共用一个应用，登录后按角色展示菜单与能力
- 管理后台：面向平台管理员、校区管理员、奖品管理员、数据管理员等角色
- 统一后端：承载账号、权限、比赛、作品、评审、积分、消息、商城、报表等服务

目标不是做一个简单报名系统，而是打通“比赛发布 -> 报名 -> 签到 -> 作品上传 -> 评审评分 -> 结果发布 -> 积分发放 -> 奖品兑换”的完整闭环。

## 2. 设计原则

### 2.1 统一账号与统一权限

所有终端共用一套用户体系、一套角色权限体系、一套组织与校区体系，避免后期出现数据割裂和账号不互通。

### 2.2 一个小程序承载学生端与老师端

老师端本质上包含学生能力的超集。首期采用同一个微信小程序，登录成功后根据身份和权限控制入口、页面与按钮，降低研发与维护成本。

### 2.3 先做核心闭环，再逐步增强

首期只追求核心价值可运行：

1. 注册登录
2. 比赛发布
3. 报名
4. 签到/作品上传
5. 评审评分
6. 结果发布
7. 积分到账

私信群聊、商城现金兑换、复杂大屏、多校区隔离、内容审核、外部系统对接均作为后续迭代。

### 2.4 按领域建模，不按页面建模

平台需要同时服务小程序和后台，后端必须按业务领域拆分模块，而不是围绕页面堆接口。

### 2.5 全链路可审计

所有关键动作都应有日志和操作记录，满足后续软著材料、问题追踪与平台运维要求。

## 3. 推荐技术方案

## 3.1 仓库组织

建议采用单仓多应用：

```text
/
├─ admin-web/                      # Vue3 管理后台
├─ miniapp/                        # uni-app 微信小程序
├─ server/                         # Spring Boot 后端
├─ infra/                          # docker-compose、初始化脚本、部署模板
├─ docs/
│  ├─ plans/
│  ├─ api/
│  ├─ db/
│  └─ runbooks/
├─ scripts/
├─ pnpm-workspace.yaml
└─ README.md
```

## 3.2 前端与后端选型

- 管理后台：Vue 3 + Vite + TypeScript + Pinia + Vue Router + Element Plus + ECharts
- 小程序：uni-app(Vue 3) + TypeScript + Pinia
- 后端：Spring Boot 3 + MyBatis-Plus + MySQL 8 + Redis
- 鉴权：Sa-Token
- 文件存储：MinIO
- 导出：EasyExcel + PDF 生成方案
- 接口文档：Springdoc OpenAPI + Knife4j
- 日志审计：Logback + 操作日志表
- 对象映射：MapStruct
- 数据校验：Hibernate Validator

## 3.3 基础设施建议

- 本地开发：Docker 启动 MySQL、Redis、MinIO
- 生产环境：Nginx + Spring Boot + MySQL + Redis + MinIO
- 监控先保留接口：首期以日志和健康检查为主，后续再接 Prometheus/Grafana

## 6. 核心业务闭环

### 6.1 核心主链路

1. 用户注册/登录
2. 老师完善资料并发布比赛
3. 学生浏览比赛并报名
4. 学生到场签到，按规则上传作品
5. 老师或评委完成评审评分
6. 老师发布排名、奖项和电子奖状
7. 系统自动发放积分并记录流水

### 6.2 首期必须保障的业务规则

- 同一学号只能注册一个账号
- 同一比赛报名人数不能超过名额
- 取消报名仅限截止前
- 签到必须符合时间窗口
- 作品上传必须符合类型、时间和大小限制
- 评分结果发布后积分只能撤回一次，且必须有日志
- 关键操作均写操作日志

## 9. 非功能设计要点

## 9.1 安全

- 密码存储使用 `BCrypt`，不按需求文档中的 `MD5 + 盐` 实现
- 全部接口走 HTTPS
- 登录、短信、签到、分享加入限流
- 所有越权访问直接拒绝并落日志
- 文件上传校验 MIME、扩展名、大小

## 9.2 性能

- 比赛列表、订单列表、积分记录等高频列表必须分页
- 热门统计和大屏数据优先读缓存
- 文件走对象存储，不进入数据库大字段
- 导出任务异步化，超大导出走后台任务

## 9.3 可维护性

- 后端采用分层：controller / app / domain / infra
- 前端统一接口层、状态层、路由守卫、权限指令
- 文档分离：接口文档、表结构文档、部署文档

## 10. 分阶段实施策略

### Phase 0：工程初始化

- 初始化仓库与多应用结构
- 准备本地开发环境
- 搭建基础 CI 脚本、代码规范、环境变量模板

### Phase 1：MVP 核心闭环

- 账号登录与个人资料
- 比赛发布与比赛浏览
- 报名、签到、作品上传
- 评审评分、结果发布
- 积分到账

### Phase 2：平台管理增强

- 后台用户/角色/校区管理
- 比赛总控
- 奖状模板
- 系统日志和轮播图

### Phase 3：积分商城与消息

- 每日签到
- 分享积分
- 奖品商城
- 兑换订单
- 系统消息
- 私信与群聊

### Phase 4：运营与高级能力

- 数据大屏
- 报表导出
- 内容审核
- 备份恢复
- 多校区隔离
- 第三方系统对接

## 11. 高风险清单

| 风险点 | 原因 | 建议 |
|--------|------|------|
| 微信小程序老师端与学生端共存 | 页面入口、权限和菜单容易混乱 | 先做统一角色路由和权限守卫 |
| 电子奖状生成 | 牵涉模板、图片合成、导出 | 首期先实现基础固定模板 |
| 群聊与私信 | 需要消息状态、会话、未读、推送 | 延后到 Phase 3 |
| 内容审核 | 需要第三方接口与异步处理 | 首期仅保留接口扩展点 |
| 多校区与复杂 RBAC | 数据隔离和权限维度很多 | 先做单校区兼容模型，第二阶段增强 |
| 大屏统计 | 聚合复杂、性能敏感 | 先做基础汇总接口，重报表后置 |

## 12. 当前建议

当前最合理的推进方式是：

1. 先完成工程初始化和 MVP 设计落地
2. 再做数据库设计与接口清单
3. 然后按“后端基础 -> 后台页面 -> 小程序页面 -> 联调 -> 验收”推进

这条路线最稳，也最适合大项目从 0 开始搭起来。
