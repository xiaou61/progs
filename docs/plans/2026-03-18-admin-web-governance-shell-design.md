# 后台管理端补齐设计

## 背景

当前 `admin-web` 已经具备大部分业务页面和 API 调用，但缺少真正可用的后台壳子。主要问题集中在四类：

1. 登录页仍是初始化占位，没有表单、状态管理、登录守卫和退出能力。
2. 应用缺少左右结构后台框架，页面都是独立直出，无法形成统一的后台管理体验。
3. 请求层没有接入登录态，页面虽然调了很多后台接口，但并未携带 token。
4. `Banner`、`系统配置`、`校区管理` 等页面仍有占位文案或未完整接入真实后端数据。

## 目标

把 `admin-web` 提升为一个可实际联调的后台管理端，具备：

- 可用的管理员登录闭环
- 左侧菜单 + 右侧内容区的标准后台布局
- 顶部标题、当前管理员信息、退出登录能力
- 路由级登录守卫
- 请求层自动携带 token
- 已有后端接口尽可能都在后台页面里接起来

## 方案对比

### 方案 A：保留现有业务页，补一层后台壳

- 优点：改动最稳，现有页面逻辑基本可复用，能最快形成可用后台。
- 缺点：部分旧页面样式会出现与新壳子风格不完全统一的问题，需要后续慢慢收口。

### 方案 B：重做整个后台 UI，再回填业务页

- 优点：视觉一致性更好。
- 缺点：成本高，风险大，会把已经能跑的业务页一起推倒重做。

### 方案 C：只补登录，不动布局

- 优点：改动最小。
- 缺点：不能解决用户实际最在意的“后台像后台”的体验问题，也无法统一菜单、守卫和导航。

## 采用方案

采用 **方案 A**。

核心原则是：

- 业务页尽量不重写，只把真正缺口补齐
- 公共问题在壳层解决，而不是每页各写一套
- 先解决“能进、能看、能管”，再慢慢做统一视觉深化

## 架构设计

### 1. 登录态

新增后台专用 `Pinia` store，维护：

- `token`
- `userId`
- `roleCode`
- `studentNo`
- `displayName`

并使用 `localStorage` 做启动恢复。登录成功后写入 store 和存储；退出时清空。

### 2. 路由结构

路由拆成两层：

- `/login`：独立登录页
- `/`：后台布局页，所有业务页作为它的子路由

所有后台业务路由都挂到布局页下，并通过 `meta` 提供：

- `title`
- `menuKey`
- `requiresAuth`

### 3. 后台布局

新增 `AdminLayout`：

- 左侧固定菜单栏
- 顶部标题和管理员信息区
- 右侧 `router-view`

左侧菜单按模块分组：

- 工作台
- 比赛管理
- 评审与结果
- 系统治理
- 内容审核

### 4. 请求层

统一在 `api/http.ts` 中注入 `Authorization: Bearer <token>`。

这样所有现有 API 模块都能自动带 token，无需逐页修改请求头。

### 5. 页面补齐

重点补齐：

- `LoginPage.vue`：真实登录表单
- `CampusListPage.vue`：补 loading/error 和真实提示文案
- `BannerPage.vue`：改成调用 `/api/admin/banners`
- `SystemConfigPage.vue`：改成调用 `/api/admin/configs`

其余已接入真实接口的页面尽量复用现状，放进统一后台壳即可。

## 风险与处理

### 1. 后端没有单独的 admin login 接口

当前后端只有 `/api/app/auth/login`，但已支持 `ADMIN` 角色和演示管理员账号。

处理方式：

- 先直接复用该接口
- 后台登录页固定以 `ADMIN` 身份登录

### 2. 登录返回没有管理员姓名

当前后端登录返回只含 `userId / roleCode / token`。

处理方式：

- 前端先保存登录时输入的账号作为展示名兜底
- 后续若后端扩展返回 `realName`，前端无缝接入

### 3. 旧页面样式分散

处理方式：

- 当前只统一外层布局和导航
- 页面内部样式后续按模块逐步收口，不阻塞可用性

## 验收标准

- 管理员可以使用演示账号完成登录
- 未登录访问后台业务路由会被重定向到登录页
- 登录后看到左侧菜单 + 右侧内容区的后台结构
- 顶部可以看到当前管理员信息，并支持退出
- 轮播图、系统配置、校区管理页不再是占位文案
- `admin-web` 测试通过，构建通过
