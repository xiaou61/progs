# 全量 Mock 清理与真实链路补齐设计

## 背景

当前仓库的大部分主链路已经接入真实接口，但仍残留若干演示预设、硬编码默认值和依赖手输内部 ID 的后台交互。这些问题会让系统在“接口已通”的表象下仍表现出明显的半成品状态。

本次改造目标不是推翻现有实现，而是在尽量复用已有接口与数据结构的前提下，系统性清理显式 mock / demo 入口，并补齐最明显的未完成交互。

## 目标

1. 移除用户可见的演示账号、默认密码、占位评委名等 mock 痕迹。
2. 将后台关键页面从“手输内部 ID”升级为“真实候选选择”。
3. 保持现有真实接口优先，不为追求完美额外引入无必要的新接口。
4. 避开工作区内正在进行的认证加固改动，减少冲突风险。

## 范围

### 1. 小程序登录

- 移除登录页中的演示账号按钮。
- 去掉默认预填学号与密码。
- 保留真实登录流程与角色切换。

### 2. 后台评审工作台

- 移除默认 `competitionId = 1`、`reviewerName = '王老师'` 等写死值。
- 改为先选择比赛，再加载评审任务与已发布结果。
- 评审老师字段默认留空，或在可解析当前管理员信息时使用真实姓名。
- 保持“从任务带入作品 / 学生”能力，但避免靠占位默认值维持流程。

### 3. 后端评审任务

- 待评任务不再返回 `"默认评委组"`。
- 对未评审记录统一返回业务空值，由前端按空态渲染。

### 4. 后台比赛发布

- 不再手输 `organizerId`。
- 通过真实用户列表筛选老师账号并供选择。
- 表单展示老师姓名、学号，提交时仍传真实 `organizerId`。

### 5. 后台报名补录

- 不再手输 `userId`。
- 通过真实用户列表筛选学生账号并供选择。
- 页面展示姓名、学号、手机号，减少误操作。

### 6. 管理端账号展示细节

- 登录后尽量展示真实姓名而非学号占位。
- 若后端登录返回尚未包含姓名，则补齐最小字段扩展。

## 非目标

1. 不移除 `demo-data` profile 下的测试 / 开发基线账号。
2. 不重做整个认证体系。
3. 不大规模重构既有后端模块结构。

## 方案选择

### 方案 A：只删演示文案

优点：

- 改动小。
- 风险低。

缺点：

- 仍然保留大量“手输 ID”的半成品体验。

### 方案 B：平衡补齐

优点：

- 能去掉大多数显式 mock。
- 能提升关键后台页面可用性。

缺点：

- 仍有一些可追求的细节留白。

### 方案 C：全量深挖

优点：

- 一轮收敛最明显的 mock、占位实现和未完成交互。
- 能把后台主链路从“接口已通”提升为“可实际运营使用”。

缺点：

- 影响前端、后端、小程序多个模块，需要更严格的测试与回归。

本次采用方案 C。

## 数据流设计

### 小程序登录

用户选择身份、输入学号密码后直接走真实登录接口。页面不再提供任何演示账号预填能力，避免误导为测试系统。

### 后台比赛发布

页面初始化时并行获取比赛列表与用户列表。从用户列表中筛出老师角色，构造成候选项。用户在页面中选择老师后，提交时仅传对应 `organizerId`。

### 后台报名补录

页面初始化时获取比赛列表与用户列表。从用户列表中筛出学生角色，构造为补录候选项。补录表单提交时传 `userId`，页面展示信息则尽量采用可读的姓名、学号、手机号。

### 后台评审工作台

页面先加载比赛列表。选择比赛后，再拉取该比赛的待评任务与已发布结果。待评任务若未分配评委，后端返回空值，前端以“未分配”或空输入框展示，而不是伪造占位姓名。

### 管理端登录显示

后台登录成功后，如果登录返回中包含真实姓名，则直接写入会话。若没有，则前端保底用学号，但本次优先补后端返回字段以改善展示。

## 错误处理

1. 老师候选为空时，比赛发布页明确提示“暂无可选老师账号”。
2. 学生候选为空时，报名补录页明确提示“暂无可补录学生账号”。
3. 待评任务未填写评委名时，不使用占位值，前端用空态提示。
4. 历史记录缺失姓名等字段时，优先做兼容展示，不阻塞页面渲染。

## 测试策略

### 前端

1. 新增或修改 `vitest` 用例，验证：
   - 后台登录页无演示账号文案。
   - 评审页无硬编码默认比赛 ID / 评审老师名。
   - 比赛发布页与报名补录页使用真实候选数据而非手输 ID。

### 后端

1. 为登录返回真实姓名补充 `MockMvc` 断言。
2. 为评审任务空评委场景补充 `MockMvc` 回归，确保不再返回 `"默认评委组"`。

### 小程序

1. 对登录页源码进行静态断言，确保不再出现演示账号按钮与默认密码。

## 风险与缓解

1. 认证相关文件当前在工作区已有未提交改动。
   - 缓解：仅最小化修改登录返回结构与消费方，不触碰 `auth/security` 新增文件。

2. 用户候选依赖现有用户接口数据完整性。
   - 缓解：复用现有 `/api/admin/users`，并在前端提供空态与兼容显示。

3. 全量改造跨前后端，容易出现字段联动回归。
   - 缓解：严格按 TDD 顺序补测试，分层验证前端和后端。
