# 真鉴权收口设计

## 背景

当前后端登录仅返回 `dev-token-*` 字符串，服务端没有实际的 token 校验链路，前后端更多依赖本地会话存储和页面守卫完成“已登录”判断。这意味着：

- 后端接口缺少真正的身份校验。
- `ADMIN`、`TEACHER`、`STUDENT` 的角色访问边界没有在服务端收口。
- 多个 app 端接口仍允许请求方传入任意 `userId`、`studentId`、`teacherId`，存在越权访问空间。

## 目标

本轮仅解决“真鉴权”问题，不同时引入文件上传、反馈治理等后续需求。

交付目标：

1. 登录接口返回可校验的签名 token。
2. 服务端拦截并校验受保护接口的 `Authorization: Bearer <token>`。
3. `ADMIN` 端接口要求管理员 token。
4. 核心 app 端接口要求登录 token，并校验请求中的用户标识与 token 中主体一致。
5. 保持前端登录链路和现有 API 结构尽量不变，减少联动改动面。

## 方案对比

### 方案 A：接入 Spring Security + JWT

优点：

- 标准化程度高。
- 后续扩展权限模型更顺手。

缺点：

- 当前项目没有 `spring-boot-starter-security`，改造面会明显扩大。
- 这轮需求更偏“尽快把现有链路做成真鉴权”，不适合一次性引入完整安全框架。

### 方案 B：自定义签名 token + MVC 拦截器

优点：

- 贴合当前工程结构，落地快。
- 不需要大改 controller/service 分层。
- 可以在本轮内顺手补上角色校验和主体一致性校验。

缺点：

- 不是完整安全框架，后续如果要扩展为更复杂的权限体系，还需要继续演进。

### 结论

本轮采用方案 B。

## 设计

### Token 结构

新增轻量 token 服务，使用服务端密钥对以下字段进行签名：

- `userId`
- `studentNo`
- `roleCode`
- `issuedAt`
- `expiresAt`

token 采用“载荷 + HMAC 签名”的形式，服务端在每次请求时校验签名和过期时间。

### 服务端拦截规则

公开接口保持免鉴权：

- `/api/public/**`
- `/api/app/auth/login`
- `/api/app/auth/register`
- `GET /api/app/competitions/**`

其余规则：

- `/api/admin/**` 仅允许 `ADMIN`
- `/api/app/**` 要求已登录用户

### 请求上下文

拦截器校验成功后，把当前登录主体放入请求上下文，供控制器读取。

主体信息至少包含：

- `userId`
- `studentNo`
- `roleCode`

### 授权校验

对存在主体绑定的 app 接口补控制器级一致性校验：

- 个人资料、密码、反馈、注销：`userId == token.userId`
- 每日任务、积分、学生结果：路径或请求体中的用户标识必须等于当前用户
- 私信、群聊、会话偏好：发送者或会话所属用户必须等于当前用户
- 教师仪表盘：`teacherId == token.userId` 且 `roleCode == TEACHER`
- 报名、取消报名、提交作品、签到：请求体中的 `userId` 必须等于当前用户

### 本轮不处理

以下内容留到后续顺序项：

- 关闭 `/api/app/auth/register`
- 去掉用户端演示账号入口
- 作品真实文件上传
- 审核规则后台配置化
- 反馈后台治理页

## 测试策略

先写失败测试，再实现：

1. 无 token 访问后台接口返回 403
2. 学生 token 访问后台接口返回 403
3. 已登录用户读取/修改他人资料返回 403
4. 教师查看其他教师仪表盘返回 403
5. 合法 token 访问原有链路仍然成功
