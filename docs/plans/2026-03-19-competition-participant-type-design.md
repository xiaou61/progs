# 比赛参赛类型与指导老师设计

## 背景

当前系统的比赛模型只有基础字段，报名也仍然是“一个用户一条记录”的个人报名模式。业务上现在需要增加更明确的参赛限制：

1. 比赛只能是“仅学生参加”或“仅老师参加”两种类型之一。
2. 比赛总人数限制继续保留，语义不变，仍然表示最多允许多少个报名人。
3. 如果是“仅学生参加”的比赛，则后台在发布比赛时必须固定绑定一名指导老师。
4. 该比赛下所有学生报名记录都默认归属于这名指导老师，不再支持学生自行选择老师，也不做队伍系统。

这意味着功能重点不是重构报名模型，而是在现有个人报名链路上补齐“参赛资格限制 + 指导老师归属信息”。

## 目标

1. 为比赛增加可配置的参赛类型。
2. 为仅学生参加的比赛强制配置一名指导老师。
3. 在报名、后台管理和展示链路中体现指导老师信息。
4. 保持现有个人报名、个人签到、个人提交作品、个人出结果的主模型不变。

## 非目标

1. 不做组队系统。
2. 不做一个比赛多位指导老师。
3. 不做学生自行选择或变更指导老师。
4. 不把作品提交、签到、评审、结果改成老师代理模式。

## 方案对比

### 方案 A：只补参赛类型校验

- 比赛新增参赛类型字段。
- 报名时只校验当前用户角色是否允许报名。

优点：

- 改动最小。

缺点：

- 无法承载“学生赛必须绑定指导老师”的业务要求。
- 后台管理也无法看出学生比赛归属哪位老师。

### 方案 B：参赛类型 + 固定指导老师

- 比赛新增参赛类型和指导老师字段。
- 报名仍是个人报名。
- 学生赛必须绑定指导老师，报名记录通过比赛派生出所属指导老师。
- 后台与小程序同步展示参赛类型和指导老师。

优点：

- 完整覆盖当前需求。
- 对现有报名和作品链路影响最小。
- 便于在后台做统计和筛选。

缺点：

- 一个比赛只能挂一位指导老师，后续如果扩成多老师需要再设计。

### 方案 C：比赛老师关系扩展表

- 引入比赛与老师的关联表。
- 一个比赛可绑定多位老师，再从报名记录或学生维度做归属分配。

优点：

- 扩展性强。

缺点：

- 远超当前需求范围。
- 会显著增加后台配置和报名逻辑复杂度。

## 结论

采用方案 B。

## 详细设计

### 一、比赛模型扩展

在比赛表新增两个字段：

1. `participant_type`
   - 枚举值：
     - `STUDENT_ONLY`
     - `TEACHER_ONLY`
2. `advisor_teacher_id`
   - 仅 `STUDENT_ONLY` 时允许有值
   - 指向系统中角色为 `TEACHER` 的账号

现有 `quota` 字段不变，继续表示总报名人数上限。

### 二、比赛发布与更新校验

在比赛保存草稿、发布比赛、更新比赛时新增校验：

1. `participant_type` 必填。
2. 当 `participant_type = STUDENT_ONLY` 时：
   - `advisor_teacher_id` 必填。
   - 对应用户必须存在。
   - 对应用户角色必须是 `TEACHER`。
3. 当 `participant_type = TEACHER_ONLY` 时：
   - `advisor_teacher_id` 必须为空。

### 三、报名校验

报名仍使用当前个人报名接口与个人报名记录模型，但校验增强为：

1. 先校验比赛时间窗口。
2. 再校验当前用户角色与比赛 `participant_type` 是否匹配：
   - `STUDENT_ONLY` 只允许学生报名。
   - `TEACHER_ONLY` 只允许老师报名。
3. 再校验 `quota`。
4. 最后校验重复报名。

### 四、返回结构与展示信息

比赛相关返回结构统一补充：

1. `participantType`
2. `advisorTeacherId`
3. `advisorTeacherName`

这样前端不必每次再额外映射老师姓名。

报名记录模型本身不新增指导老师字段，指导老师通过比赛信息派生展示即可。

### 五、后台页面

#### 比赛发布页

新增“参赛类型”配置区：

1. 单选：
   - 仅学生参加
   - 仅老师参加
2. 当选择“仅学生参加”时，显示指导老师下拉：
   - 仅列出启用状态的老师账号
3. 当选择“仅老师参加”时，自动清空指导老师选择

#### 报名管理页

在比赛摘要和报名记录区域补充展示：

1. 比赛参赛类型
2. 指导老师姓名（仅学生赛显示）

### 六、小程序页面

#### 比赛列表 / 详情页

展示：

1. 参赛类型标签
2. 指导老师信息（仅学生赛显示）

#### 报名页

报名前直接显示：

1. 当前账号角色
2. 当前比赛允许的参赛类型
3. 指导老师信息（学生赛）

如果角色不匹配，则禁止报名并给出明确提示。

### 七、兼容策略

历史比赛数据没有这两个字段时，迁移后需要补齐默认值，建议：

1. 旧数据默认按 `STUDENT_ONLY` 处理或通过脚本回填。
2. 若存在旧的学生赛但没有指导老师，需要在迁移脚本或管理后台中补录后才能继续编辑发布。

实现时更稳妥的做法是：

1. 数据库层先允许 `advisor_teacher_id` 为空。
2. 业务层对新建、更新、发布进行严格校验。
3. 历史数据只要不重新编辑，可保持只读兼容。

## 测试策略

### 后端

1. 学生赛保存时未选择指导老师应失败。
2. 学生赛保存时选择非老师账号应失败。
3. 老师赛保存时带指导老师应失败。
4. 学生账号不能报名老师赛。
5. 老师账号不能报名学生赛。

### 后台前端

1. 比赛发布页切换参赛类型时应正确显示/隐藏指导老师下拉。
2. 保存草稿和发布时应携带新的参赛字段。
3. 报名管理页应展示参赛类型和指导老师。

### 小程序

1. 比赛详情和报名页应展示参赛类型与指导老师。
2. 角色不匹配时报名动作应被阻止。

## 风险与规避

1. 历史比赛数据没有指导老师。
   - 规避：仅对新建、编辑、发布时强校验，避免直接阻断旧数据读取。

2. 比赛接口结构变更会影响后台和小程序多处页面。
   - 规避：统一扩展比赛摘要/详情模型，减少前端二次拼装。

3. 指导老师选择依赖用户列表中的角色信息。
   - 规避：后台前端继续复用现有用户接口，后端保存时再做最终角色校验，避免仅依赖前端过滤。
