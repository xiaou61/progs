# 本地文件上传与按比赛查看结果设计

## 背景

当前微信小程序的学生主链路还残留两个明显的半成品点：

1. 作品上传页仍然要求用户手动填写文件 URL，而不是直接选择本地文件上传。
2. 比赛详情页跳转到结果页时虽然会携带 `competitionId`，但结果页并没有解析该参数，而是始终展示学生的全部比赛结果。

这两个问题会让“查看结果”和“上传作品”两条核心链路在接口已通的前提下仍然表现出明显的样板性质。

## 目标

1. 小程序作品上传统一改为“选择本地文件并上传到服务器本地目录”。
2. 结果页支持两种模式：
   - 从比赛详情进入时，只显示当前比赛结果；
   - 从首页或菜单进入时，继续显示学生全部结果。
3. 复用现有数据结构，不引入额外的云存储或第三方对象存储依赖。

## 非目标

1. 不引入七牛、OSS、COS 等外部文件服务。
2. 不改动后台评审或审核流程对 `fileUrl` 的业务含义，仍保持其为可访问的文件地址。
3. 不重做整套认证加固逻辑，避免与当前工作区里在途的 auth/security 改动冲突。

## 方案对比

### 方案 A：真实本地上传到服务器目录

- 后端新增 multipart 上传接口，把文件保存到服务端本地目录。
- 服务端通过静态资源映射暴露 `/uploads/submissions/**` 访问路径。
- 小程序使用 `wx.chooseMessageFile` 选择文件，再用 `wx.uploadFile` 上传，成功后将返回的 URL 写入提交流程。

优点：

- 用户体验最接近真实产品。
- 后台、评审、审核仍然能拿到真实文件地址。
- 后续如果要接对象存储，替换存储服务即可。

缺点：

- 需要补一个本地文件存储服务和静态资源映射。

### 方案 B：前端伪上传，仍提交本地路径

- 小程序选择文件后直接把本地路径当作 `fileUrl` 提交。

优点：

- 实现最快。

缺点：

- 服务端和后台无法访问手机本地路径。
- 评审、审核、回显都会失效。

### 方案 C：继续手填 URL，只补结果页过滤

- 只解决结果页参数失效问题，不处理上传链路。

优点：

- 风险最小。

缺点：

- 作品上传仍然是假流程。

## 结论

采用方案 A。

## 详细设计

### 一、文件上传后端

新增一个独立的提交文件上传控制器，避免直接改动当前已被用户在途修改的提交控制器。

- 路径建议：`POST /api/app/submission-files`
- 请求方式：`multipart/form-data`
- 字段：`file`
- 返回结构：
  - `fileName`
  - `fileUrl`
  - `size`

文件统一写入本地目录：

- 物理目录：`server/local-storage/submissions`
- 访问路径：`/uploads/submissions/<generated-name>`

命名策略：

- 用时间戳或 UUID 生成新文件名，保留原扩展名。
- 原始文件名只用于展示，不直接作为落盘文件名，避免重名和非法字符问题。

错误处理：

- 空文件直接报错。
- 没有原始文件名时报错。
- 文件写入失败时返回统一业务异常。

### 二、静态资源映射

新增一个单独的 Web MVC 配置类，只负责暴露上传目录：

- `/uploads/submissions/**` -> `file:<absolute-path>/`

这样既不需要改动用户在途的 `WebMvcConfig.java`，也能避免和认证拦截配置互相覆盖。

### 三、小程序作品上传

新增上传服务方法，封装 `wx.uploadFile`。

页面流程改为：

1. 点击“选择本地文件”
2. 调用 `wx.chooseMessageFile`
3. 选中文件后显示文件名和大小
4. 点击“上传并提交作品”
5. 先上传文件拿到 `fileUrl`
6. 再调用现有提交接口 `/api/app/submissions`

页面状态新增：

- `selectedFileName`
- `selectedFileSize`
- `uploading`
- `selectedFilePath`

回显规则：

- 若已有历史版本，仍显示服务端返回的 `fileUrl`
- 当前新选择但尚未提交的文件，优先显示本地文件名，不直接覆盖版本列表

### 四、结果页按比赛过滤

结果页新增 `competitionId` 解析能力：

- `competitionId > 0` 时，仅从学生结果集中筛出当前比赛结果
- `competitionId = 0` 时，继续展示全部结果

同时修正登录回跳：

- 若结果页带了 `competitionId`，则 `requireLogin()` 的 redirect 也要带上参数

这样从比赛详情点击“查看结果”后，即使先跳登录，再回来也不会丢失当前比赛上下文。

## 测试策略

### 后端

新增上传接口测试：

1. multipart 上传成功后返回 `fileUrl`
2. 本地目录中实际生成文件
3. 访问路径以 `/uploads/submissions/` 开头

### 小程序

新增或扩展 Node 测试：

1. 上传服务应通过 `wx.uploadFile` 上传并正确解析 `ApiResponse`
2. 结果页应根据 `competitionId` 过滤结果
3. 结果页无参数时应展示全部结果

## 风险与规避

1. 当前工作区已有未提交的 auth/security 改造。
   规避：上传相关实现不直接修改这些文件，尽量新增独立控制器与独立资源配置。

2. Windows 本地路径和静态资源映射容易出现斜杠问题。
   规避：统一使用 `Path.toAbsolutePath().toUri().toString()` 生成资源目录 URI。

3. 小程序本地文件 API 在不同平台返回结构略有差异。
   规避：只依赖 `name`、`size`、`path` 这几个最常见字段，并在页面中做缺省兜底。
