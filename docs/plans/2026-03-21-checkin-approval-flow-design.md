# 签到申请制与老师确认流设计

## 背景

当前比赛签到存在两套彼此独立的状态：

1. 小程序签到会写入 `cmp_checkin`，所以学生端/老师端自己看起来已经“签到成功”。
2. 后台报名管理页展示的是报名表 `cmp_registration.attendance_status`，只有管理员手动点“标记到场”才会变成“已到场”。

这会导致前台显示“已签到”，后台却依然是“待签到”，业务含义也和“必须老师确认后才算到场”的规则不一致。

## 目标

1. 将比赛签到改成“提交签到申请 -> 老师确认/驳回 -> 最终到场状态”的单一路径。
2. 小程序签到不再直接等于到场，只表示“已提交申请”。
3. 老师后台确认后，报名记录才真正变成 `PRESENT`。
4. 定位权限本轮不做，不作为签到前置条件。

## 方案对比

### 方案 A：签到申请制

- 小程序点击签到后，仅写入一条状态为 `PENDING` 的签到申请。
- 后台查看报名记录时同时看到签到申请状态。
- 老师点击“确认签到”后，签到申请变为 `APPROVED`，报名记录 `attendanceStatus` 变为 `PRESENT`。
- 老师点击“驳回申请”后，签到申请变为 `REJECTED`，报名记录保持 `PENDING`。

这是本次采用的方案。它最符合“必须老师确认后才算到场”的业务规则。

### 方案 B：前台直签到，后台仅可撤销

- 学生一提交就直接变成到场。
- 老师后台只能事后改回。

不采用。因为这和用户刚确认的规则冲突。

### 方案 C：必须定位或扫码后才能签到

- 把定位、二维码作为签到前置条件。

本轮不采用。因为现场容错成本高，而且当前最核心问题是状态链路不一致，不是验证方式不够严。

## 详细设计

### 一、签到状态模型

`cmp_checkin` 从“纯签到记录”升级为“签到申请记录”，新增字段：

- `status`：`PENDING` / `APPROVED` / `REJECTED`
- `reviewRemark`：老师驳回说明
- `reviewedAt`：老师处理时间

并约束同一比赛、同一用户只保留一条签到申请记录，重复申请时走更新而不是重复插入。

### 二、小程序状态

小程序签到页不再只看有没有签到记录，而是综合：

- 报名记录里的最终到场状态 `attendanceStatus`
- 签到申请记录里的申请状态 `checkin.status`

展示规则：

- 无申请 + 到场待定：`待提交`
- `PENDING`：`待老师确认`
- `APPROVED` 或报名已到场：`已确认到场`
- `REJECTED`：`申请已驳回，可重新提交`
- 报名已缺席：`已缺席`

按钮规则：

- 待提交 / 已驳回：允许“提交签到申请”
- 待老师确认：按钮禁用，显示“已提交申请”
- 已确认到场 / 已缺席：按钮禁用

### 三、后台状态

后台报名管理页继续以报名记录为主体，但列表数据要合并签到申请信息，新增展示：

- 签到申请状态
- 申请提交时间
- 驳回原因

老师操作规则：

- `确认签到`：将申请状态改成 `APPROVED`，并同步把报名状态改成 `PRESENT`
- `驳回申请`：将申请状态改成 `REJECTED`，并保留驳回原因
- `标记缺席`：仍保留后台手动缺席能力；如果用户已有签到申请，则同步把申请改成 `REJECTED`

### 四、接口设计

后端新增一个老师确认接口，推荐挂在报名管理域下：

- `POST /api/admin/registrations/{registrationId}/checkin-review`

请求体：

- `status`: `APPROVED` / `REJECTED`
- `reason`: 驳回原因，可选；`REJECTED` 时必填

同时，后台报名列表接口返回增强后的管理视图，不直接复用前台简版报名 DTO。

### 五、兼容策略

- 现有前台报名接口仍可保留原字段，不强行让所有调用方都处理签到申请字段。
- 后台管理列表改成单独的管理视图 DTO。
- 已存在的“手动标记到场/缺席”能力保留，但会同步修正签到申请状态，避免两边再次分叉。

## 测试策略

1. 后端新增 API 集成测试，覆盖：
   - 小程序提交签到申请后状态为 `PENDING`
   - 后台报名列表可看到待确认状态
   - 老师确认后报名记录变成 `PRESENT`
   - 老师驳回后前台可重新提交
2. 后台新增 API 客户端测试，覆盖签到审核接口。
3. 小程序新增源码测试，覆盖“待老师确认”“申请已驳回”“提交签到申请”等关键文案。
