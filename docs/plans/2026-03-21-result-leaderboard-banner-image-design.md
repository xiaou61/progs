# 比赛榜单公开与图片轮播设计

## 背景

当前小程序比赛结果页走的是“学生个人结果”接口，因此只能看到当前账号自己的获奖结果，无法查看某个比赛已经发布的完整获奖榜单。与此同时，首页轮播图虽然已经接入了后台配置，但后台数据结构只支持标题、状态与跳转路径，无法上传和展示图片。

## 目标

1. 比赛结果页进入某个比赛后，展示该比赛全部已发布结果，让用户可以看到谁获得了一等奖、二等奖等。
2. 首页轮播图升级为图片轮播，后台支持直接上传本地图片文件。
3. 轮播图跳转路径改为可选，留空时只展示图片，不执行跳转。

## 方案选择

### 方案 A：直接把比赛结果页改成比赛榜单，轮播图补图片上传

这是本次采用的方案。

- 小程序结果页继续使用当前路由，不新增页面。
- 当带 `competitionId` 进入时，直接请求比赛维度的结果接口，而不是个人结果接口。
- 后端比赛结果接口补充获奖人姓名与学号，前端用真实身份信息展示榜单。
- 轮播图表结构新增 `image_url` 字段，后台提供图片上传接口与预览，首页优先显示图片。

优点：

- 改动集中，入口不变，用户最容易理解。
- 不需要新增“榜单页”和“个人成绩页”双入口。
- 图片轮播与后台配置保持一致，运营修改成本低。

缺点：

- 需要补充 banner 图片上传存储链路和数据库迁移。

### 方案 B：保留个人结果页，再新增比赛榜单页

优点是角色区分更清晰；缺点是页面和入口更多，本轮收益不如方案 A。

### 方案 C：轮播图继续用文字卡，只加图片 URL 输入框

开发更快，但后台老师仍然要先自己找外链地址，和“直接上传本地图片文件”的诉求不一致，因此不采用。

## 详细设计

### 1. 比赛结果页

- 小程序 `pages/competition/result/index` 在带 `competitionId` 时改调比赛榜单接口。
- 页面展示内容改为：
  - 获奖人姓名
  - 学号
  - 奖项
  - 名次
  - 成绩
  - 评审意见
  - 发布时间
- 电子奖状编号不再对所有人公开，只在榜单中保留奖项结果信息，避免把个人证书编号完全暴露给其他用户。
- 如果后端返回空列表，则仍显示“当前还没有已发布的比赛结果”。

### 2. 比赛榜单接口

- 继续使用 `/api/app/results/competition/{competitionId}` 这条路由，不新增额外 URL。
- 返回类型从 `ScoreSummary` 提升为“比赛榜单视图”：
  - 保留原有成绩、奖项、名次、发布时间
  - 新增 `studentNo`、`studentName`
- 服务层通过 `UserMapper` 将成绩记录中的 `studentId` 映射为姓名与学号。

### 3. 轮播图图片上传

- `sys_banner` 新增 `image_url` 字段。
- 新增独立的 banner 文件存储服务，文件落到单独目录并通过 `/uploads/banners/**` 暴露。
- 后台新增 banner 图片上传接口，使用 multipart/form-data。
- 后台轮播图管理页新增：
  - 当前图片预览
  - 上传本地图片按钮
  - 图片为空时的占位提示
- 轮播图保存时：
  - `title` 仍保留，作为后台识别和小程序图片缺省说明
  - `jumpPath` 可为空
  - `imageUrl` 可为空，但首页会优先使用图片，缺图时回退为文字卡片

### 4. 小程序首页轮播图

- 首页 `swiper` 优先渲染图片。
- 如果某条 banner 没有 `imageUrl`，则沿用当前文字卡片样式，避免旧数据不可见。
- 点击轮播项时：
  - `jumpPath` 有值：正常 `navigateTo`
  - `jumpPath` 为空：不跳转，不报错

## 测试策略

- 后端：
  - 比赛结果接口单测，验证榜单返回姓名和学号
  - banner 图片控制器/存储单测，验证图片 URL 返回
- 管理端：
  - API client 测试补 `imageUrl`
  - 轮播图页面源码测试补上传与预览存在性
- 小程序：
  - 结果页改用比赛榜单接口的源码/工具测试
  - 首页轮播图图片渲染与“空跳转路径可展示”的源码测试

## 风险与处理

- 旧 banner 数据没有图片：前端保留文字兜底。
- 结果榜单展示姓名属于业务公开范围：本轮按用户要求开放比赛结果查看。
- 轮播图跳转路径留空：前端显式短路，避免点击报错。
