# Findings & Decisions

## Requirements
- 项目是一个校园师生比赛管理平台，包含学生端微信小程序、老师端微信小程序能力、管理后台、Spring Boot + MySQL 后端。
- 文档明确要求覆盖比赛全生命周期：发布、报名、签到、作品上传、评审、评分、排名、奖状、积分发放。
- 平台还包含积分商城、订单管理、消息私信/群聊、数据大屏、权限管理、多校区管理等重模块。
- 明确要求支持导出 Excel/PDF/图片、富文本、文件预览、电子奖状、内容审核、系统监控、日志审计。
- 非功能要求较高，包含并发、性能、安全、备份、扩展性、多端适配等。

## Research Findings
- 需求文档产品名称为“师生比赛管理微信小程序”，定位是一体化校园比赛数字化管理平台。
- 核心参与角色至少包括学生、老师、管理员，且管理员侧需支持自定义角色和细粒度权限。
- 业务域不止“比赛”，还至少包括用户、组织、多校区、作品、评审、积分、奖品、订单、消息、通知、统计报表、系统配置。
- 老师端不仅是“发布比赛”，还包含评审、评分、积分发放、群消息、个人数据大屏等复杂能力。
- 学生端不仅是“报名比赛”，还包含签到、上传作品、奖状查看、积分任务、商城兑换、私信互动。
- 这是典型的“平台型系统”，不适合一步到位开发全量功能，必须按业务闭环分阶段推进。
- 第一批实现任务适合先做工程骨架和后端基础，因为这部分对后续三端都有承载作用。
- 第二批实现任务已经证明：`admin-web` 适合直接走 Vite + Vue3，`miniapp` 需要走 `uni` CLI 而不是直接调用 `vite`。
- 当前 MVP 已证明：用“内存态服务 + SQL 迁移脚本占位”的方式，可以先把比赛主链路与积分链路跑通，再平滑切到 MySQL 持久化。
- 当前持久化升级也证明：先从 `auth`、`competition`、`registration` 这类核心依赖域开始切库，能以最小风险带动后续领域迁移。
- 当前第二轮持久化升级证明：`checkin`、`submission`、`score`、`points` 也可以沿用“双通道服务”模式平滑切库，而不破坏已有手工单测。
- 评分结果不仅要保存名次和分数，也必须显式持久化积分值，否则学生结果查询会出现“已发积分但结果页显示 0”的数据不一致问题。
- 当前前端联调证明：先从“只读型核心页面”切真实接口最稳妥，既能快速看到真实业务数据，也不会一上来就把复杂表单流全压到同一批次。
- 当前表单型联调证明：在后端命令模型还比较精简时，前端应明确展示“当前账号确认提交”而不是伪造不会入库的表单字段，这样链路更真实也更可维护。
- 当前详情链路联调证明：先把“列表进入详情，再从详情进入动作页”打通，比在列表页直接堆多个按钮更清晰，也更符合小程序操作路径。
- 当前闭环入口联调证明：老师发布比赛和后台发布结果这两个入口一旦打通，系统就已经具备完整的“创建比赛 -> 学生参与 -> 发布结果”操作闭环，只差端到端验收。
- 当前真实登录联调证明：如果没有开发环境演示账号，老师端和学生端的真实体验仍会被“无账号可登”卡住，所以登录路径和演示数据必须一起补齐。
- 当前范围重划证明：一旦明确“不接第三方”，就必须同步从计划里彻底移除短信、支付、服务通知、外部审核、外部系统对接等能力，否则后续实现顺序会持续失焦。
- 当前个人中心批次证明：账号资料、密码、反馈、注销这类用户基础能力适合优先于消息和后台治理落地，因为它们会直接影响后续消息开关、隐私权限和违规处理模型。
- 当前全量测试暴露出：测试用例不能把数据库主键当成稳定值，尤其在混合 profile、演示数据和全量回归场景下，应优先通过业务唯一键定位测试对象。
- 当前比赛管理批次证明：一旦比赛进入真实联调阶段，“只能发布不能管理”会很快成为瓶颈，所以草稿、编辑、推荐/置顶、下架必须尽早补齐。
- 当前比赛公开列表也需要和管理列表分流：前台只应该看到 `PUBLISHED`，而后台和老师端必须看到 `DRAFT / PUBLISHED / OFFLINE` 全量状态。
- 当前报名管理批次证明：比赛进入执行期后，学生端取消报名、后台驳回/补录、到场/缺席标记必须和同一张报名表联动，否则签到、作品、结果链路会出现状态分叉。
- 当前评审增强批次证明：评审意见不应该只停留在老师端工作台，而应直接跟随结果发布进入学生结果页，这样“评审 -> 结果 -> 奖状”才是一个完整闭环。
- 电子奖状首版不必一开始做复杂模板引擎，只要先保证编号、标题和获奖记录可稳定生成并可追溯，就能满足当前本地可运行版验收目标。
- 当前每日任务批次证明：每日签到和比赛分享完全可以复用 `pts_record` 做“同类任务每日一次”的限制，不需要额外再造一张任务表。
- 当前个人比赛汇总批次证明：我的比赛、作品、获奖、积分汇总应直接聚合报名、作品、结果和积分账户四类真实数据，而不是再维护一份独立统计快照。
- 当前消息中心批次证明：系统消息、私信和比赛群聊完全可以先基于站内消息模型本地闭环，不需要引入 WebSocket 或第三方消息网关，也能满足 PRD 的首版可运行目标。
- 咨询老师入口最适合直接挂在比赛详情页上，而不是先做独立客服页；这样用户问题产生的上下文最明确，也更利于把私信场景和比赛主链路联动起来。
- 当前后台治理批次证明：角色管理和用户治理动作不必等完整 RBAC 框架全部到位后再做，先把角色数据、冻结/解冻、重置密码、改角色做成真实页面，就能快速把后台从“只读”推进到“可治理”。
- 用户治理页比“每行堆一排按钮”更适合采用“左侧列表 + 右侧动作面板”的结构，这样角色切换、冻结原因和密码重置能复用同一份选中上下文，联调和后续扩展都更稳。
- 当前数据大屏批次证明：后台汇总看板和老师个人看板不需要再额外维护统计快照表，直接聚合比赛、报名、作品、成绩四类真实数据就足够支撑首版本地运行验收。
- 导出首版不必强依赖第三方 Excel 方案，CSV 已足够覆盖“本地可运行、可验收、可复核”的当前目标。

## Technical Decisions
| Decision | Rationale |
|----------|-----------|
| 推荐采用“统一后端 + 管理后台 Web + 单个微信小程序按角色分流”的总体形态 | 降低首期研发与运维复杂度，统一账号、权限和业务数据 |
| 业务设计按领域拆分，而不是按页面拆分 | 该项目涉及多端协同，按领域建模更利于复用和扩展 |
| 首期优先打通“登录鉴权 -> 比赛发布 -> 报名 -> 签到/作品 -> 评审评分 -> 结果发布 -> 积分入账”主链路 | 这是平台成立的最小核心价值闭环 |
| 消息、积分商城、群聊、大屏、导出、多校区建议后置分阶段上线 | 复杂度高、依赖多，适合作为第二阶段及以后迭代 |
| 执行顺序先做 `Task 1-3` | 能快速建立仓库结构、后端启动能力和统一返回规范 |
| `miniapp` 安装依赖时必须绕开 workspace 模式 | 当前盘符对 `pnpm` 链接方式不友好，隔离安装更稳定 |
| 账号基础域首版先用“内存实现 + SQL 迁移占位” | 先打通行为测试，后续再接 MyBatis-Plus 和真实数据库 |
| 比赛、签到、作品、评分、积分、运营能力本轮继续沿用“内存实现 + 迁移脚本占位” | 保持 TDD 节奏，优先交付可验证的 MVP 闭环 |
| 持久化升级先采用“Spring Bean 走数据库、手工 new 的单测保留内存分支”的兼容方案 | 能在不中断现有测试体系的情况下逐步把领域迁移到真实数据库 |
| `score.points` 必须与结果发布同时落库到 `cmp_score` | 避免结果查询与积分流水之间出现数据不一致，保证学生端结果页可直接展示积分奖励 |
| 前端先统一抽象 HTTP 请求层，再逐页替换静态数据 | 可复用 `ApiResponse` 解包、错误处理与环境基址，减少后续页面接入成本 |
| 表单型页面优先围绕现有后端命令模型设计交互 | 避免前端先采集一堆后端不会保存的数据，降低后续联调偏差 |
| 小程序主链路优先采用“列表 -> 详情 -> 动作页”的跳转结构 | 降低页面信息密度，让报名、签到、上传、结果查看都有清晰入口 |
| 老师端和后台的关键入口要先具备真实提交能力 | 这样后续做端到端联调时不需要再回头补“入口页还是骨架”的问题 |
| 开发环境需要内置最小演示账号集 | 让老师/学生两种身份都能不依赖手工造数直接登录体验完整流程 |
| 小程序真实登录打通后，还必须补齐登录态持久化与登录回跳 | 否则每次重启或跳转都会重新卡在登录入口，端到端联调体验不连续 |
| 比赛主链路的自动化联调用例不能写死绝对时间窗口 | 随着真实日期推进，报名/签到/提交会被业务规则拦截，测试应围绕“相对当前时间”的合法场景构造数据 |
| 剩余 PRD 模块要先做“去第三方裁剪”再进入实现 | 否则任务清单会持续夹杂无法在本地闭环的能力，影响验收标准 |
| 个人中心首批接口继续沿用现有真实登录和用户表扩展 | 可以在不打断主链路的前提下补齐资料、密码、反馈、注销四类能力 |
| 个人中心自动化测试应按学号查用户而不是写死主键 | 演示账号和全量测试顺序都会影响自增 ID，学号才是稳定业务标识 |
| 比赛管理增强需要把“公开列表”和“管理列表”拆成两套查询语义 | 学生端只看已发布比赛，老师端和后台必须能管理草稿和下架记录 |
| 草稿和发布复用同一张比赛表最合适 | 现有 `status` 字段已足够承载 `DRAFT / PUBLISHED / OFFLINE`，只需扩展推荐/置顶和更新时间字段即可 |
| 报名管理增强继续复用 `cmp_registration` 最合适 | 取消、驳回、补录、到场/缺席都属于同一参赛记录生命周期，没必要拆第二套流程表 |
| 评审增强继续复用 `cmp_review_task` 最合适 | 评审任务本身已经承载比赛与作品关系，只需扩展学生、评审意见、建议分数字段即可 |
| 电子奖状首版采用“固定模板标题 + 自动编号”方案 | 当前目标是打通本地可运行闭环，先保证结果发布时能稳定生成奖状标识，后续再扩展视觉模板 |
| 结果发布时同步固化评审老师和评审意见 | 避免老师侧后续改评语导致学生历史结果展示漂移，保持结果快照语义稳定 |
| 每日签到和比赛分享继续复用 `pts_record` 最合适 | 已有积分流水已具备用户、业务类型、业务 ID 和时间维度，足够支持“每日一次”的任务判定 |
| 个人比赛汇总优先按实时聚合实现 | 当前数据量小、场景偏本地可运行版，直接聚合报名/作品/结果/积分数据比额外维护汇总表更简单可靠 |
| 消息中心首版采用“系统消息表 + 用户会话表 + 消息明细表”三表模型 | 既能支持系统消息、私信、比赛群聊三类场景，也能在不引入实时推送的前提下维护未读数和最新消息 |
| 比赛群聊成员首版不单独维护成员表 | 当前已能从比赛发起人和有效报名记录实时推导群成员，足够支撑本地可运行版闭环 |
| 后台治理首版采用 `sys_role + sys_user.role_code` 的简化模型 | 当前目标是先交付可运行的角色管理和用户治理闭环，不急于引入角色-菜单-按钮三层关系表 |
| `@MapperScan` 必须限制为只扫描带 `@Mapper` 注解的接口 | 否则像 `AdminUserService` 这样的服务接口会被误注册为 MyBatis Mapper，导致运行期绑定错误 |
| 数据大屏首版采用“实时聚合 + CSV 导出”方案 | 当前数据量小，直接从真实业务表聚合比维护快照或引入 Excel 库更轻、更稳、更适合本地可运行版本 |
| 本地审核首版采用“敏感词常量 + 作品文件白名单 + 违规记录表”方案 | 能覆盖当前 PRD 里最关键的审核闭环，又不需要引入任何第三方审核服务 |

## Issues Encountered
| Issue | Resolution |
|-------|------------|
| 需求文档内容非常大且覆盖范围远超常规毕业设计/中小项目 | 采用“领域拆分 + 分阶段交付”的方式控制范围 |
| 当前仓库为空，没有现成代码上下文 | 先输出架构与计划，后续再初始化多端工程 |
| 并行读取 `docx_requirements.md` 时因执行先后导致首次读取失败 | 先确认文件落盘，再执行后续搜索与章节校验 |
| `java -version` 在当前 PowerShell 下不可直接调用 | 使用 `mvn -v` 确认 Maven 已绑定 JDK 17，暂不影响后端构建 |
| Maven 测试输出存在 `rdc` profile 警告 | 该警告来自本机 Maven 配置，不影响当前项目测试通过 |
| Git 在当前盘符下提示 `dubious ownership` | 暂不改全局配置，待需要提交时再处理 `safe.directory` |
| `pnpm` 默认 workspace 安装在当前盘符会触发链接类错误 | `miniapp` 采用 `--ignore-workspace` 和 copy 导入方式安装成功 |
| `uni-app` 插件默认导出并非直接函数 | `vite.config.ts` 里兼容处理 `default` 导出后恢复正常 |
| PowerShell 会把 Maven `-Dtest=CheckinServiceTest,SubmissionServiceTest` 这类参数拆开 | 为 `-Dtest` 参数整体加引号后，Maven 可正常识别多个测试类 |
| 新增业务测试初次失败有一部分来自测试数据窗口不合法，而不是功能实现错误 | 统一把用例前置时间修正为“报名未截止、比赛已开始”的合法状态 |
| 引入持久化后，SpringBoot 测试需要显式使用可迁移的测试数据库 | 使用 H2 MySQL 模式 + Flyway 后，新增落库测试和原有集成测试均可正常运行 |
| `ScorePointsPersistenceTest` 暴露出数据库分支未保存积分值 | 增加 `V6__score_points_column.sql`，并同步修正 `ScoreEntity` 与 `ScoreService` 的持久化映射 |
| 前端静态骨架页一旦切成真实接口，请求失败会直接暴露在页面上 | 页面必须补齐 `loading / empty / error` 三态，否则联调时很难区分“无数据”和“请求失败” |
| 报名页最初的姓名/学号/手机号表单并不会进入当前后端命令 | 调整为“当前账号确认报名”模式，并把解释直接展示在页面上 |
| `uni-app` 测试并行执行时会争用本地端口 | 小程序测试改为串行执行，避免 `EADDRINUSE` 噪音干扰结果判断 |
| 老师端发布页和后台结果发布入口都还是骨架时，主链路无法真正闭环 | 分别补齐真实提交表单、校验和刷新逻辑后，闭环入口已经可用 |
| 登录页仍是骨架时，真实老师/学生入口虽然存在但无法从正常路径进入 | 增加开发环境演示账号初始化和登录页接线后，角色入口已经可通过真实登录进入 |
| 小程序如果只把 token 存在内存 store，重启后会丢失登录态 | 增加本地存储恢复和基础登录守卫后，真实联调可以连续完成 |
| 接口级闭环测试若使用固定未来日期，报名接口会按规则拒绝 | 改成以 `LocalDateTime.now()` 为基准生成合法窗口后，整条链路可以稳定回归 |
| Windows 下一次性补大块补丁容易触发命令长度限制 | 改为分段 `apply_patch` 后，页面、样式和文档都能稳定落地 |
| `ProfileApiTest` 初版依赖固定 `userId=2`，在全量测试里会误命中老师账号 | 改为按 `studentNo` 动态查找学生 ID 后，全量后端测试恢复稳定 |
| 原有比赛发布页只覆盖“新建”场景，导致老师无法编辑存量比赛 | 增加管理列表和选中回填后，草稿、编辑、下架链路都能在同一页完成 |
| 执行剩余模块计划时，Git worktree 方案会因仓库当前全部为未提交状态而无法安全承接现有上下文 | 本轮继续沿用当前工作区开发，不做空白 worktree 切换 |
| 评审增强首轮接口测试命中了不存在的 `/api/admin/reviews/submit` 路由 | 补齐控制器映射和服务后恢复为标准业务接口 |
| 每日任务首轮接口测试命中了不存在的 `/api/app/points/tasks/*` 路由 | 补齐任务控制器和相关模型后恢复为标准业务接口 |
| Windows 下消息模块的首次整包补丁写入再次触发命令长度限制 | 切换为骨架优先、分段 `apply_patch` 的 staged write 方案后恢复稳定 |
| 收紧 `@MapperScan` 之后，历史上未显式标注 `@Mapper` 的 `FeedbackMapper` 无法被 Spring 注入 | 给 `FeedbackMapper` 显式补上 `@Mapper`，并以此作为后续 Mapper 编写约束 |
| `MockMvc` 手工读取审核接口响应体时，如果未显式带出 UTF-8 charset，会导致中文断言出现伪失败 | 为审核接口显式声明 `application/json;charset=UTF-8` 后，红绿灯恢复稳定 |
